Benchmark option flags currently can only be configured via command-line arguments (for example, passing `--benchmark_out=<filename>`). This makes it difficult to integrate benchmark execution into environments where configuration is commonly injected via environment variables (such as CI systems). The library should also support reading option defaults from environment variables, using a `BENCHMARK_` prefix, while still letting explicit command-line flags override anything coming from the environment.

Implement initialization of benchmark option flags from environment variables when they are defined. For example, setting `BENCHMARK_OUT=<filename>` must behave the same as specifying `--benchmark_out=<filename>`, and setting `BENCHMARK_OUT_FORMAT=json` must behave the same as `--benchmark_out_format=json`. If both an environment variable and a command-line flag are provided for the same option, the command-line value must take precedence.

In addition, boolean options read from the environment must be parsed consistently via a helper like `BoolFromEnv(name, default_value)`. When the corresponding environment variable `BENCHMARK_<NAME_IN_UPPERCASE>` is not set, `BoolFromEnv` must return `default_value`. When it is set, it must recognize common truthy/falsey strings case-insensitively. The following values must be treated as false: `0`, `n`, `no`, `f`, `false`, `off` (and their case variations). The following values must be treated as true: `1`, `y`, `yes`, `t`, `true`, `on` (and their case variations). The behavior should be portable across platforms including Windows, where setting/unsetting environment variables differs.

After this change, users should be able to configure benchmark execution purely through environment variables, without changing command invocations, and still be able to override any such configuration by providing explicit `--benchmark_*` flags on the command line.
