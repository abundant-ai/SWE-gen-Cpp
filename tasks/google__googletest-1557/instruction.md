Building code that uses Google Mock’s `NiceMock`, `NaggyMock`, or `StrictMock` wrappers can trigger undefined behavior during construction. The problem comes from the internal base used by these wrappers (the CRTP-style `internal::{Nice,Naggy,Strict}Base<MockClass>`) attempting, inside its constructor, to treat `this` as if it were already a `({Nice,Naggy,Strict}Mock<MockClass>)` object (i.e., casting a base pointer to a derived pointer while the derived object is not yet fully constructed). This is undefined behavior in C++ and can be caught by Control Flow Integrity (CFI) instrumentation, causing builds to fail or abort when constructing these wrappers.

The construction of `NiceMock<T>`, `NaggyMock<T>`, and `StrictMock<T>` must not perform any cast from the internal base subobject to the derived wrapper type during base-class construction. Creating instances such as `NiceMock<MockFoo> mock;` (and similarly `NaggyMock<>`/`StrictMock<>`, including for mocks without default constructors when forwarding arguments) should be safe under CFI and must not rely on object layout assumptions during construction.

Expected behavior: wrapping a mock type `T` with `NiceMock<T>`, `NaggyMock<T>`, or `StrictMock<T>` should be well-defined C++ with no UB in constructors, should work with forwarded constructor arguments, and should preserve the intended semantics (Nice = uninteresting calls allowed, Naggy = warn on uninteresting calls, Strict = fail on uninteresting calls).

Actual behavior: the internal base constructor performs a downcast to the wrapper type before the wrapper is fully formed, which is UB and can be rejected at runtime by CFI builds.

Fix this by restructuring the `NiceMock`/`NaggyMock`/`StrictMock` class hierarchy so that construction and registration of the mock’s “uninteresting call” behavior does not require base-to-derived casting during base construction, while keeping the public API (`testing::NiceMock`, `testing::NaggyMock`, `testing::StrictMock`) and behavior intact.
