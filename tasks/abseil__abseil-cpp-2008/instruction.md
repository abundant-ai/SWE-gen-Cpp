`absl::HexStringToBytes()` incorrectly decodes some hex inputs due to sign-extension when converting the parsed hex value into the output byte buffer. This can cause bytes with the high bit set (0x80–0xFF) to be written incorrectly, producing different results depending on platform/compiler and whether `char` is signed.

When converting a hex string to bytes, each pair of hex digits must map to exactly one byte with the same numeric value. For example, calling `absl::HexStringToBytes("80")` should produce a 1-byte string whose only byte has value `0x80`, and `absl::HexStringToBytes("ff")` should produce a 1-byte string with value `0xFF`. Currently, these cases can be corrupted (e.g., by interpreting intermediate values as signed and then sign-extending before storing), so round-tripping through hex escaping/unescaping utilities may fail for arbitrary byte sequences.

Fix `absl::HexStringToBytes()` so that decoding is byte-accurate for all 256 possible byte values and is independent of the signedness of `char`. The function should correctly handle mixed inputs that include any bytes in the full range, such that conversions like `absl::BytesToHexString(...)` / hex-escape generation followed by `absl::HexStringToBytes(...)` reproduce the original bytes exactly, including for bytes >= 0x80.
