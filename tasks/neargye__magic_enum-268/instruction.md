Using magic_enum with enums whose underlying values span a very large numeric range (for example, values near the limits of the underlying integer type, or a customized enum_range with a large [min,max]) currently fails to work correctly. Common operations like enumerating values/names, building compile-time containers of enum entries, and performing flags-related enumeration either become unusably slow/large at compile time, fail to compile due to excessive template/constexpr work, or produce incorrect/empty results.

The library should support “big range” enums such that the public API remains usable when enum values are sparse across a large domain. In particular, when a user customizes an enum via magic_enum::customize::enum_range<E> (including settings like min/max and is_flags), the following should behave correctly even if the numeric range between min and max is large:

- magic_enum::enum_values<E>() and magic_enum::enum_names<E>() should return the correct set of actual enumerators, not attempt to iterate every integer in the entire min..max span in a way that causes compilation failure/timeouts.
- magic_enum containers utilities (the APIs provided by magic_enum_containers.hpp) should be able to create compile-time containers keyed by enum values without requiring a contiguous small range; operations that map enum -> index (or build arrays/bitsets/lookup tables) must not overflow and must remain correct for large underlying values.
- For flag enums (enum_range<E>::is_flags = true), combining flags with bitwise operators and enumerating valid flag values should still work when individual flag bits are high (e.g., values like 1u<<N for large N within the underlying type), and any internal indexing or sizing must not assume small bit positions.

Expected behavior: For an enum E with a very large customized range and/or large enumerator values, calls like magic_enum::enum_values<E>(), magic_enum::enum_entries<E>(), and the container helpers for E should compile successfully and return the correct values in a stable order, without integer overflow or attempting to instantiate work proportional to the entire numeric span.

Actual behavior: With such enums, compilation may fail or take extremely long, and/or the returned values/containers are incorrect (missing entries, empty results, wrong indexing), indicating that the implementation assumes a small contiguous range or uses range size calculations that break for large domains.
