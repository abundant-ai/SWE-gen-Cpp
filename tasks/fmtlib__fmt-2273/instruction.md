The library currently uses a custom error type and exception types (e.g., `fmt::error_code`, `fmt::system_error`, and `fmt::windows_error`) instead of the standard C++11 facilities. This causes friction when integrating with code that expects `std::error_code` / `std::system_error`, and it prevents seamless formatting and propagation of standard error codes.

Update the OS/error-handling interfaces so that `fmt::system_error` and `fmt::windows_error` are replaced by the standard `std::system_error`, and any internal/custom `fmt::error_code` usage is replaced with `std::error_code`.

After the change:

- Throwing a system-related exception from fmt’s OS utilities should throw `std::system_error` (not a fmt-specific exception type). Code that catches `const std::system_error&` must be able to observe the correct error value via `e.code().value()`.

- Formatting a `std::error_code` with `fmt::format` must produce a stable, category-qualified representation.
  - For `std::error_code(42, std::generic_category())`, formatting with `{}` should produce `"generic:42"`.
  - For `std::error_code(42, std::system_category())`, formatting with `{}` should produce `"system:42"`.
  - For `std::error_code(-42, std::system_category())`, formatting with `{}` should produce `"system:-42"`.
  - The same must work for wide-character formatting (i.e., formatting into a `std::wstring`), producing `L"generic:42"`, `L"system:42"`, and `L"system:-42"` respectively.

- On Windows-specific conversion utilities (e.g., `fmt::detail::utf16_to_utf8`), when an invalid conversion is triggered, the thrown exception must be catchable as `std::system_error`, and its `.code().value()` must match the Windows error code used for the failure (e.g., `ERROR_INVALID_PARAMETER`). The `.what()` message must include the formatted Windows error text generated by fmt’s Windows error formatter for that error code.

The goal is that users can rely entirely on standard-library error code/exception types while still getting correct fmt formatting and correct error propagation from fmt’s OS layer.
