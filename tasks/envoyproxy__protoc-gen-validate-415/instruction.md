The Go code generator in protoc-gen-validate needs to support a `module=<module-path>` parameter (similar to `protoc-gen-go`’s `module` option). Today, when generating Go validation code for protos whose `go_package` is a fully-qualified module import path (e.g. `option go_package = "github.com/acme/project/pkg/grpc";`), users cannot reliably place the generated validation files under an arbitrary output directory while keeping the generated file paths and Go import paths aligned. This becomes especially problematic when using tools that set plugin `output: .` and rely on `module=...` to “strip” the module prefix from the output path.

When invoking the validate plugin with Go generation, it should accept a parameter like `lang=go,module=github.com/acme/project`. With this parameter set, the generator must treat the value as a Go module path prefix: when computing where to write generated `.pb.validate.go` files, it should strip the provided module prefix from the Go import path derived from `go_package` (or equivalent import path resolution), and then use the remaining suffix as the relative directory beneath the plugin output directory.

For example, given `option go_package = "github.com/acme/project/pkg/grpc"` and plugin options `output: .` with `lang=go,module=github.com/acme/project`, the generated validation file should be written under `./pkg/grpc/...` (not under `./github.com/acme/project/pkg/grpc/...`), and the generated code’s package/import behavior must still correspond to the `go_package` settings so that imports between generated packages work.

The behavior must also work when a proto embeds or references types from another Go package (different `go_package`), ensuring that cross-package embedding/validation generation succeeds with correct import paths and without duplicating the module prefix in output paths.

If `module` is not provided, existing behavior should remain unchanged. If `module` is provided but does not match the beginning of the computed Go import path for a file, generation should not incorrectly strip unrelated path segments (it should either leave the path unchanged or handle the mismatch in a predictable, non-destructive way rather than producing malformed output paths).
