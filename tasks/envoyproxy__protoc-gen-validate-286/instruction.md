C++ code generated by protoc-gen-validate does not correctly enforce the `pattern` constraint using RE2 semantics, even though PGV patterns are defined to use RE2 syntax. This is most visible when using RE2 features that are not compatible with `std::regex` (such as inline flags), and it also fails for patterns that include a null byte (`\x00`) because the pattern string is treated as null-terminated and gets truncated.

When a message contains validation rules like `pattern: "^[\\x00-\\x7f]+$"` (intended to accept ASCII bytes including 0x00), the generated C++ validator currently compiles or applies the regex using a string representation that may stop at the first `\x00`. As a result, the effective regex becomes incomplete (for example only `^[`), leading to incorrect validation behavior: values that should be rejected may be accepted, and/or valid values may be rejected, depending on the truncated prefix.

Implement RE2-based pattern validation in generated C++ validators so that `pattern` rules are compiled and evaluated using RE2, not `std::regex`, and so that patterns containing `\x00` are preserved exactly (no truncation) when constructing the RE2 pattern. This behavior must work consistently for:

- string fields with `pattern`
- bytes fields with `pattern`
- repeated fields where the `items` rule includes `pattern`
- map fields where `keys` and/or `values` rules include `pattern`

Expected behavior:

- If a field has a `pattern` rule, validation succeeds only when the entire field content matches the RE2 regex as specified by the pattern string.
- RE2 syntax is supported (including constructs/flags expressed in the pattern string as RE2 expects).
- Patterns containing escaped hex bytes (including `\x00`) are compiled and matched without truncation or loss of data.
- On mismatch, validation returns/produces the usual PGV validation failure for a pattern violation (with the correct field identified), rather than silently passing or failing due to a malformed/truncated regex.

Actual behavior:

- C++ pattern validation either uses a regex engine incompatible with PGV’s RE2 syntax and/or incorrectly constructs the regex string.
- Patterns that include a null byte are truncated when turned into a C++ string for regex compilation, causing incorrect validation results.
