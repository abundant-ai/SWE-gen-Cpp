When integrating {fmt} into older or macro-heavy C/C++ codebases, compilation can fail or produce diagnostics due to two compatibility issues.

1) Null pointer constant warnings/errors: Some code paths in {fmt} use a literal 0 (or otherwise trigger integer-literal null pointer constants) in contexts where a null pointer is intended. In projects built with strict diagnostics such as -Wzero-as-null-pointer-constant (often promoted to errors), this results in build failures. {fmt} already provides FMT_NULL as an abstraction for a proper null pointer constant, but it is not used consistently. Update the library so that places representing a null pointer use FMT_NULL consistently, eliminating -Wzero-as-null-pointer-constant warnings.

2) Enumerator name clashes with legacy macros: Some {fmt} enum enumerator names can collide with pre-existing macros/defines from legacy headers (e.g., X11-style headers defining None as 0). If a macro like None is defined before including <fmt/format.h>, compilation can break due to the macro replacing identifiers used by {fmt}. The library should compile cleanly even when common legacy macros (such as defining None to 0) are present before including {fmt} headers. Adjust the public-facing API/types so that such macro substitutions do not cause compilation errors and do not change the intended semantics of formatting.

Expected behavior: A project that defines legacy macros like `#define None 0` before including {fmt} headers should still compile; and compiling {fmt} (and typical user code that includes it) with -Wzero-as-null-pointer-constant should produce no warnings for {fmt} headers/sources. Actual behavior: compilation fails or emits these diagnostics due to inconsistent null pointer constants and macro/enumerator collisions.
