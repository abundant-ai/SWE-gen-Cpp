C++ code generated by protoc-gen-validate does not correctly support string validation rules for hostnames, IPv4 addresses, and IPv6 addresses. As a result, users cannot reliably validate configuration fields that may contain either a hostname or an IP address (a common “address” use case), and downstream code can produce confusing failures when invalid values slip through (for example, later parsing can fail with errors like "stoi" when a host:port string is malformed because the host portion wasn’t validated as a hostname or IP).

When a protobuf string field is annotated with validation rules intended to accept hostnames and/or IP addresses, the generated C++ Validate/ValidateOrThrow logic should correctly recognize valid hostnames, valid IPv4 literals, and valid IPv6 literals, and reject invalid inputs with a clear validation failure rather than letting later parsing fail.

Implement/repair the C++ validators so that the following behaviors hold when validating a string value:

- Hostname validation accepts typical DNS hostnames (labels separated by dots, composed of alphanumerics and hyphens with proper label boundaries) and rejects malformed hostnames.
- IPv4 validation accepts standard dotted-quad IPv4 literals and rejects out-of-range octets, wrong segment counts, and malformed forms.
- IPv6 validation accepts valid IPv6 literals (including common compressed forms such as "::" usage) and rejects malformed IPv6 strings.
- The generated validation should be consistent with the semantics of the existing PGV string rules (i.e., if a rule says “this must be an IPv4”, it must not accept hostnames; if it says “this must be a hostname”, it must not accept IPs; and the underlying building blocks must be correct so an eventual “address = hostname OR ip” rule can be built on top).

If an input is invalid for the selected rule(s), validation should fail deterministically (returning a failed status / false, or throwing—depending on the generated API mode) with a meaningful validation error message indicating the value does not satisfy the expected hostname/IPv4/IPv6 constraint, rather than crashing or producing unrelated exceptions.

A minimal reproduction example should work in C++:

- Define a message with a string field annotated to require an IPv4 address; validating "192.168.0.1" succeeds and validating "999.1.1.1" fails.
- Define a message with a string field annotated to require an IPv6 address; validating "2001:db8::1" succeeds and validating "2001:::1" fails.
- Define a message with a string field annotated to require a hostname; validating "example.com" succeeds and validating "-bad.host" fails.

These fixes are a prerequisite for adding a higher-level “address” rule (hostname OR IP) and should ensure C++ behavior matches the intended PGV rule semantics used by other languages.
