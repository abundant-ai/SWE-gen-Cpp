When handling a multipart/form-data request in a server handler like `[](const httplib::Request& req, httplib::Response& res)`, uploaded files and regular form fields are not easily accessible in a clean, Go-like way. In particular, when a multipart request contains both file parts and non-file parameters, it should be possible to read all non-file fields directly from `req.params`, while accessing uploaded files separately from `req.files`. Currently, form fields that arrive as multipart parts may not be represented in `req.params` (or are mixed with file parts in a way that makes it hard to retrieve the values), forcing users to manually walk multipart items and distinguish “file vs parameter” themselves.

Update the multipart parsing and request population logic so that:

- For `Content-Type: multipart/form-data` requests, every non-file form field part (i.e., a part with a `name` but without an uploaded filename/content-disposition file attribute) is added to `httplib::Request::params` the same way URL query parameters are.
- Every uploaded file part is added to `httplib::Request::files` (keyed by the form field name), and files should not pollute `req.params`.
- This must work when a single multipart request contains a mix of files and non-file fields (“file_and_param” scenario). For example, if the client posts a multipart body containing fields like `token=abc` and a file part named `image`, the handler should be able to do `req.get_param_value("token")` (or look up `req.params`) and separately find `req.files["image"]` with the uploaded file metadata and content.

Ensure consistency with existing request APIs: `req.params` should behave like it does for query strings (including supporting multiple values for the same key if that is already supported by the library), and `req.files` should continue to provide access to `httplib::MultipartFormData` / `FormData` entries including `name`, `filename`, `content_type`, and `content` (or equivalent fields). The change should not break existing non-multipart request handling.

If a user attempts to retrieve a multipart file via parameter APIs (e.g., treating a file part as a param), it should not appear as a normal parameter value. Conversely, non-file multipart fields must be retrievable through the normal parameter access methods, without requiring manual iteration over multipart items.
