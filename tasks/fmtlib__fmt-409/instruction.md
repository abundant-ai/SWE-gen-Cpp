Building projects that include fmt headers with GCC’s `-Wzero-as-null-pointer-constant` enabled produces large numbers of warnings because fmt code uses the literal `0` as a null pointer constant in declarations, defaults, and assignments (e.g., pointer variables initialized to `0`, default pointer parameters set to `0`, and pointer fields reset to `0`). This is especially problematic because fmt is heavily template-based and these warnings originate from template instantiations in user translation units, so users cannot realistically suppress the warnings only for fmt.

The library should provide a single portable null-pointer macro, `FMT_NULLPTR`, that expands to `nullptr` when the compiler supports C++11 null pointers and otherwise expands to an acceptable null pointer constant for older compilers. All public headers and internal code that currently uses a literal `0` to mean “null pointer” should use `FMT_NULLPTR` instead.

After this change, compiling user code that includes fmt headers with `-Wzero-as-null-pointer-constant` should not emit warnings from fmt due to `0` being used as a null pointer constant.

This should also work in environments where `windows.h` is included before fmt headers, and in environments where macros like `None` may be defined to `0` by third-party headers. The change must not alter runtime behavior: pointer members and pointer return/argument semantics should remain identical, and code such as an `AllocatorRef` type that stores an `Allocator*` should still allow default construction to a null allocator, moving should still null out the moved-from allocator pointer, and calls that forward pointer arguments should still accept and forward a null pointer value.

In addition, any places where virtual function overrides are intended but older compilers require a compatibility annotation should use the project’s override-compatibility macro (`FMT_OVERRIDE`) where needed so compilation remains clean across supported compilers.
