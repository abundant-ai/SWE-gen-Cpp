When establishing a WebSocket connection, Crow currently requires the server to choose a subprotocol value to return in the `Sec-WebSocket-Protocol` response header. In some deployments, clients send `Sec-WebSocket-Protocol` with a value that is not predictable or not known to the server ahead of time, but the server must still echo something back for browser clients to accept the connection.

The problem is that Crow does not provide a way to “mirror” the incoming `Sec-WebSocket-Protocol` header value back in the handshake response. As a result, if the application cannot preselect the exact subprotocol string, the server either omits the header or sends a different value, and browser clients fail the WebSocket handshake due to subprotocol mismatch.

Add support so that a WebSocket route/handler can opt into mirroring the received `Sec-WebSocket-Protocol` header value back to the client during the upgrade response. When the client includes `Sec-WebSocket-Protocol` in the request, and mirroring is enabled, the server’s handshake response must include `Sec-WebSocket-Protocol` with exactly the same value as received (byte-for-byte, preserving the full header value). If the client does not send `Sec-WebSocket-Protocol`, the server must not add the header.

This should work for typical browser handshakes where the client includes the header during `Upgrade: websocket` and expects the negotiated subprotocol to be echoed back. The change must not break existing behavior where applications explicitly set a chosen subprotocol; mirroring should be an additional, opt-in capability.
