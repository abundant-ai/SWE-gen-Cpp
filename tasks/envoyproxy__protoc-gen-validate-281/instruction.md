C++ code generated by protoc-gen-validate does not fully validate map fields. In particular, validations are not being applied to the values of map entries, which can allow invalid embedded messages to pass validation.

A common failure mode is a configuration message that contains a map from string to a message type (e.g., a policy object). If the policy message contains embedded fields with constraints (for example repeated fields requiring at least one element via `min_items = 1`, or required message fields), `Validate()` should fail when those constraints are violated inside the map value. Instead, validation succeeds, and downstream code may crash when it assumes the constraints were enforced.

For example, consider a message with a field like:

```proto
map<string, Policy> policies = 2;
```

If a `Policy` instance stored in `policies["foo"]` has invalid contents (such as an empty repeated field that is constrained to have at least one item), calling validation on the outer message should return an error (or throw, depending on the validation API used) indicating the nested violation. Currently, validation incorrectly reports success, letting invalid configs proceed.

The generator/runtime must be updated so that, for C++ validation:

1) Map-level constraints are enforced for map fields, including `min_pairs` and `max_pairs`.

2) `no_sparse` is enforced for maps whose value type is a message: if `no_sparse = true`, the map must not contain entries whose message value is unset/empty in a way that indicates a missing value.

3) Key/value rules are enforced:
- If `(validate.rules).map.keys` is set, each key must be validated against the specified key rules.
- If `(validate.rules).map.values` is set, each value must be validated against the specified value rules.

4) Recursive validation is performed for message-typed map values even when no explicit `(validate.rules).map.values` is provided. If the map value type is a message that has its own generated `Validate()` logic, that validation must be invoked for every map entry’s value. This should catch nested constraints like `string.min_len`, `repeated.min_items`, required message rules, etc.

As a concrete recursive example, if you have:

```proto
message MapRecursive {
  map<uint32, Msg> val = 1;
  message Msg {
    string val = 1 [(validate.rules).string.min_len = 3];
  }
}
```

then validating `MapRecursive` should fail when the map contains any entry whose `Msg.val` is shorter than 3 characters.

Expected behavior: calling the generated C++ validation (e.g., `Validate()` / `ValidateOrThrow()` depending on the API surface in use) on a message containing a map must reject invalid map sizes, invalid keys, invalid values, and invalid nested message contents within map values, producing a validation error message that points to the offending field/entry.

Actual behavior: nested constraints inside map values are skipped, so invalid map values can pass validation and cause crashes later in consumers that rely on validated invariants.
