Crow is vulnerable when a client uses HTTP pipelining (sending multiple HTTP requests back-to-back on the same TCP connection without waiting for prior responses). Under certain timing/orderings, Crow can delete/close a connection after finishing a request, but still later execute code that assumes the connection object is alive (use-after-free). This can manifest as a crash, memory corruption, or other undefined behavior when handling pipelined requests.

Reproduction scenario: open a single TCP connection to the server and write two HTTP/1.1 requests in a single write (or quickly one after another) so they are received as a pipeline. If the first request causes the server to close the connection (or otherwise triggers connection teardown), Crow must not continue parsing/dispatching the next pipelined request using the now-destroyed connection state. Expected behavior is that pipelined requests are not processed once the connection is scheduled for deletion/closure; Crow should either (a) safely serialize/stop processing additional buffered requests when closing, or (b) reject/disable HTTP pipelining such that only one in-flight request is processed per connection and any extra bytes are ignored/lead to a safe close.

Additionally, Crow currently allows WebSocket connections to be established even if the HTTP request is invalid in a basic way: a WebSocket upgrade request without a Host header can succeed. This is incorrect for HTTP/1.1, where Host is mandatory. When attempting a WebSocket handshake with an invalid HTTP/1.1 request that does not include a Host header, the upgrade must be rejected (i.e., it must not switch protocols to WebSocket).

Fix both issues so that: (1) HTTP pipelining can no longer lead to execution after the connection object has been destroyed (no use-after-free), and (2) WebSocket upgrade requests are validated as proper HTTP/1.1 requests, including requiring a Host header, before establishing the WebSocket connection.
